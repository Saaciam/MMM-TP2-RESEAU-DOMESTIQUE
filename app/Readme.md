1. Le Patron de Conception MVVM (L'Architecture)Votre projet est divisé en trois couches distinctes pour respecter les standards professionnels d'Android :•Le Modèle (Graph, RoomPlan, NetworkObject) : Ce sont de simples classes de données. Elles ne savent pas comment s'afficher à l'écran. Elles stockent uniquement les informations (ex: "Il y a un mur à la case [10,5]").•La Vue (DrawView, MainActivity) : C'est ce que l'utilisateur voit. La DrawView est responsable de transformer les données mathématiques en pixels colorés sur le Canvas.•Le ViewModel (NetworkViewModel) : C'est le cerveau. Il contient les instances du Graphe et du Plan. Sa force est qu'il survit à la rotation de l'écran. Si vous tournez le téléphone, l'Activity meurt, mais le ViewModel reste en vie, préservant ainsi votre réseau en cours de création

2. Le Système de Coordonnées (Le "Monde" vs l' "Écran")C'est la partie la plus mathématique de votre code. Il y a deux référentiels :1.L'Écran (View) : Les pixels fixes sur votre téléphone. Le point (0,0) est toujours le coin en haut à gauche de l'écran.2.Le Monde (Le Plan) : C'est votre appartement de 2000x2000 pixels.Le mécanisme de Navigation :•offsetX / offsetY : Ces variables stockent de combien vous avez "poussé" le plan.•canvas.translate(offsetX, offsetY) : Dans onDraw, on dit à Android : "Déplace ta feuille de dessin de X et Y avant de commencer à dessiner les murs".•Calcul Inverse (realX, realY) : Dans onLongPress, si l'utilisateur clique à l'écran au point (100, 100) mais que vous avez décalé le plan de 50 pixels, l'objet doit être créé au point (150, 150) du "Monde". D'où la formule : e.x - offsetX.

3. La Matrice de Collision (RoomPlan)Au lieu d'utiliser une simple image, vous utilisez une Grille (Matrix).•Chaque case fait cellSize (60 pixels).•Si grid[row][col] == 1, c'est un mur.•Pourquoi ? Cela permet une logique binaire très rapide : pour savoir si on peut poser un objet à la position (X, Y), on divise X et Y par 60 pour trouver la case correspondante et on vérifie si elle est égale à 1.

4. La Gestion des Gestes (GestureDetector)Android ne devine pas tout seul ce qu'est un "glissement" ou un "appui long".•onScroll : Cette fonction est appelée continuellement quand vous déplacez votre doigt. On récupère la distanceX (le mouvement du doigt) pour mettre à jour nos offsets.•limitScroll() : C'est la fonction de sécurité. Elle calcule si le bord du plan blanc va dépasser le bord de l'écran. Si c'est le cas, elle bloque l' offset à une valeur maximum ou minimum. C'est ce qui donne l'effet de "butée" quand vous arrivez au bout de l'appartement.•onTouchEvent : C'est le point d'entrée. Sans le gestureDetector.onTouchEvent(event), votre détecteur serait comme une oreille bouchée : il ne recevrait jamais les informations tactiles.

5. Le Dessin (onDraw)La méthode onDraw est appelée à chaque fois que vous faites invalidate().1.canvas.save() : On prend une "photo" de l'état vierge du canvas.2.Navigation : On applique le déplacement.3.Dessin du Sol : On dessine un grand rectangle blanc.4.Dessin des Murs : On boucle sur la matrice. Si on trouve un 1, on dessine un rectangle gris.5.canvas.restore() : On revient à l'état vierge pour ne pas perturber les prochains dessins du système Android.

Points clés à retenir pour votre examen/TP :1.Pourquoi le ViewModel ? Pour la persistance des données lors de la rotation.2.Pourquoi le Canvas ? Pour avoir un contrôle total sur le dessin graphique (murs, connexions courbées, etc.) que les composants standards XML ne permettent pas.3.Pourquoi coerceIn ? Pour simplifier le code des limites de scroll (évite les multiples if/else).4.L'importance de cellSize : C'est votre unité de mesure. Si vous la passez à 100, tout votre appartement s'agrandit proportionnellement.
